# Recipe: Workspace Onboarding

> An AI agent without a workspace map is like a new employee on their first day
> with no office tour: they'll figure it out, but they'll waste everyone's time doing it.

## The Problem

When an AI agent (Claude, Codex, Gemini, Copilot) opens a session on your workspace
for the first time, it doesn't know:

- How many repos you have, or where they are
- What each repo does
- What language/framework each uses
- Your branch strategy, PR rules, deploy targets
- Where secrets live, what connections exist

**Without this map, every agent burns its first minutes scanning the filesystem.**

We learned this the hard way on EasyWay: a 14-repo polyrepo where Codex routinely
spent 8+ exploratory queries before writing a single line of code. The solution was
`factory.yml` — a single file that answers every orientation question. But that file
took 148 sessions to evolve. This recipe gives you the same result in 15 minutes.

## The Pattern: Bootstrap Sequence

```
Agent opens session
  → looks for workspace-map.yml in known location
  → found?  YES → load it (cost: 1 file read, ~0s)
             NO → trigger onboarding interview
                  → generate workspace-map.yml
                  → all future sessions cost zero
```

This is the **Electrical Socket Pattern** (G16): standard interface (`workspace-map.yml`),
interchangeable provider (interview, scan, manual edit).

## Prerequisites

- A workspace with one or more repos
- An AI agent that can read/write files
- 15 minutes of your time (interview mode) or 5 minutes (scan mode)

## Option A: Interview Mode (recommended for first-timers)

Use this when there's no existing documentation. The agent asks you questions,
one at a time, and builds the map from your answers.

### Interview Flow

Inspired by the [Valentino Design Preference Questionnaire](https://dev.azure.com/EasyWayData/EasyWay-DataPortal/_git/easyway-portal?path=/docs/valentino/design-preference-questionnaire.md),
this interview follows the same principles:

- **One question at a time** — never dump the whole list
- **Multiple-choice when possible** — reduce cognitive load
- **Separate facts from preferences** — the map captures structure, not opinions

#### Opening

```
I need to build a map of your workspace so I (and any other AI agent)
can orient instantly in future sessions. I'll ask you 5-8 short questions.
The result is a workspace-map.yml file that any agent can read.
```

#### Questions (5 core + 3 optional)

**Core (always ask):**

1. **Repos**: How many repositories are in this workspace? Can you list them?
   - Example: "3 repos: backend, frontend, infra"
   - For monorepos: "1 repo with apps/backend, apps/frontend, packages/"

2. **Stack per repo**: For each repo, what's the primary language/framework?
   - Example: "backend = Node.js/Express, frontend = React/Next.js, infra = Docker + bash"

3. **Branch strategy**: How do you merge code?
   - feat → main (simple)
   - feat → develop → main (gitflow)
   - trunk-based (commit to main)
   - mixed (different per repo)

4. **Deploy**: Where does your code run?
   - Cloud provider (AWS/GCP/Azure/Oracle)
   - Self-hosted server
   - Serverless
   - Not deployed yet

5. **Source of truth**: Where do you track work?
   - GitHub Issues
   - Azure DevOps Boards
   - Jira
   - Linear
   - Nothing formal

**Optional (ask if time allows):**

6. **Secrets**: Where do you store API keys and credentials?
7. **CI/CD**: Do you have automated pipelines? Where?
8. **Other agents**: Which AI tools does your team use? (Claude, Copilot, Cursor, Codex...)

### Output

The agent generates `workspace-map.yml` from the answers (see schema below).

## Option B: Scan Mode (for documented workspaces)

If your workspace already has documentation (README, package.json, docker-compose.yml),
the agent can scan and generate the map automatically:

```bash
# Agent scans these signals:
# - package.json / pyproject.toml / go.mod / Cargo.toml → stack
# - .git/config → remotes
# - docker-compose.yml → services
# - .github/workflows/ or azure-pipelines.yml → CI/CD
# - .cursorrules / AGENTS.md → existing agent context
```

The agent builds a draft `workspace-map.yml` and asks you to confirm/correct.

## Schema: workspace-map.yml

```yaml
# workspace-map.yml — Generated by Testuggine Onboarding
# This file is the single source of truth for workspace orientation.
# Any AI agent should read this FIRST before exploring the codebase.
#
# Origin: this schema was born from EasyWay's factory.yml (14 repos, 149 sessions,
# 40+ agents). We distilled it to the minimum an agent needs to orient instantly.

version: "1.0"
generated: "2026-03-15"
method: "interview"  # or "scan" or "manual"

workspace:
  name: "MyProject"
  description: "Brief description of what this workspace is"
  root: "."  # relative path from where this file lives

repos:
  backend:
    path: ./backend              # relative to workspace root
    stack: node                  # node | python | go | rust | java | dotnet | mixed
    framework: express           # optional: express, django, gin, etc.
    branch_strategy: feat-main   # feat-main | feat-develop-main | trunk
    pr_target: main              # default PR target branch
    deploy: true
    deploy_target: "docker"      # docker | serverless | static | none
    description: "REST API + business logic"

  frontend:
    path: ./frontend
    stack: node
    framework: nextjs
    branch_strategy: feat-main
    pr_target: main
    deploy: true
    deploy_target: "static"
    description: "Web UI"

# Optional sections (add as needed)

work_tracking:
  platform: github              # github | ado | jira | linear | none
  project_url: ""               # optional

secrets:
  location: ".env (gitignored)"  # or "vault", "cloud KMS", "server file"
  notes: "Never commit .env files"

agents:
  - name: claude
    context_file: CLAUDE.md      # or .cursorrules, AGENTS.md
  - name: copilot
    context_file: .github/copilot-instructions.md

ci_cd:
  platform: github-actions       # github-actions | ado-pipelines | gitlab-ci | none
  config_path: .github/workflows/
```

## Where to Put workspace-map.yml

| Location | When |
|----------|------|
| Workspace root (`./workspace-map.yml`) | Default — agents look here first |
| `~/.easyway/workspace-map.yml` | If workspace root is cluttered |
| Pointed from AGENTS.md / .cursorrules | Always — so agents find it instantly |

## Multi-Agent Output: Write Where Each Agent Reads

**Critical lesson from EasyWay S149**: generating `workspace-map.yml` is not enough.
Each AI agent reads instructions from a **different location**. If you only write
AGENTS.md, Codex won't see it. If you only write .cursorrules, Claude won't see it.

The onboarding must write to **all agent entry points**:

| Agent | Entry Point | What to write |
|-------|------------|---------------|
| Claude Code | `CLAUDE.md` (workspace root) | Full instructions |
| Cursor | `.cursorrules` (repo root) | Full instructions |
| Codex (OpenAI) | `~/.codex/instructions.md` | Compact instructions (global) |
| Copilot | `.github/copilot-instructions.md` | Compact instructions |
| Gemini | Project-level settings | Via IDE config |

### What goes in each file

All files should contain the same **core information**, adapted to format:

1. **Workspace map** — which repos, where, what stack
2. **ADO/Git workflow** — how to create PRs (tool, not CLI)
3. **Branch strategy** — per-repo target branches
4. **Absolute rules** — what to NEVER do (secrets, direct push, etc.)
5. **Branch prefix** — each agent uses its own prefix (claude/, codex/, cursor/)

### Template for agent-specific instructions

```markdown
# {Project} — {Agent} Instructions

## Workspace Map
{table of repos with path, stack, PR target}

## Rules
- NEVER {forbidden actions}
- ALWAYS {required workflow}

## PR Workflow
{exact commands, not generic guidance}

## Branch Prefix
Use `{agent-name}/` prefix for all branches.
```

### Codex-specific: config.toml for MCP

If you use MCP servers, also update `~/.codex/config.toml`:

```toml
[mcp_servers.your-ado-tool]
args = ["mcp/index.ts", "--transport", "stdio"]
command = "npx"
cwd = "/path/to/ado-toolkit"
```

This ensures Codex can create PRs via MCP instead of falling back to `az repos pr create`.

## Integration with Testuggine

Once `workspace-map.yml` exists, the [Governance Testuggine](governance-testuggine.md)
can add a Layer 1 rule:

```
Rule: workspace-map.yml exists and is < 30 days old
Score: +1 if present, 0 if missing, WARN if stale
```

### Layer 1 rule: multi-agent coverage

```
Rule: instructions exist for all active agents
Check: for each agent in agent-census.json,
       verify that the corresponding entry point file exists
Score: +1 per agent covered, WARN if any agent missing
```

This closes the loop: onboarding generates the map, Testuggine keeps it fresh.

## From EasyWay's Experience

This recipe distills lessons from 149 sessions of multi-agent development:

- **Session S121**: First time we noticed agents wasting time on orientation
- **Session S127**: Created `factory.yml` as the master workspace map
- **Session S147**: Context Diet reduced MEMORY from 1342 to 728 lines — but the
  workspace map was still not in the agent entry point
- **Session S149**: Codex spent 75s + 8 tool calls finding the portal repo.
  Then it tried `az repos pr create` instead of `ado-remote.sh`, committed on the
  wrong branch, and couldn't create a PR because MCP ADO wasn't configured.
  Root cause: AGENTS.md existed, but Codex doesn't read AGENTS.md — it reads
  `~/.codex/instructions.md`. The fix: write instructions in **every agent's format**.

The key insight: **the cost of NOT having a map compounds with every session**.
A 15-minute investment in onboarding saves hours across all future agent sessions.
The second insight: **writing instructions in only one format is writing them for only one agent**.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Map is too detailed (100+ lines) | Keep it under 50 lines — agents need orientation, not documentation |
| Map is in a wiki but not in the repo | Put it where agents look: workspace root or pointed from AGENTS.md |
| Map was generated once and never updated | Add Testuggine Layer 1 rule for staleness |
| Each agent has its own map format | Use this standard schema — one map, all agents |
| Instructions only in AGENTS.md | Codex, Copilot don't read it — write to each agent's entry point |
| MCP servers not configured for all agents | Check config.toml (Codex), settings.json (Cursor), etc. |
| No branch prefix convention | Agents create conflicting branches — assign prefixes (claude/, codex/, cursor/) |
